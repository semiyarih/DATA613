NAMESPACE

NAMESPACE is a file that determines

  1. What functions are available to the user of your package (versus what functions are for internal use only), and

  2. What functions from other packages are you importing.

Do NOT Edit By Hand. Means do not open the folder directly and edit it. It will be generate and edit by {usethis} package.

DESCRIPTION

DESCRIPTION includes metadata about your package, such as the package name, version, author, and dependencies.

Open DESCRIPTION file

  • Directly: Click on the file and select the text editor of your choice

  • Rstudio:

    • file.edit("DESCRIPTION"): This function opens the file in RStudio’s editor. The file is now ready to edit.

    • file.show("DESCRIPTION"): If you want to open a file in RStudio for viewing without the risk of editing it by mistake.

{roxygen2} Documentation

Documentation is vital for

  1. Maintaining packages (We may forget what our functions do)

  2. Having other folks use our package (they need a way to learn the functions).

  • You should be writing documentation while you are writing R code

  • Documentation in an R package is in “.rd” files in the “man” folder.

{roxygen2}

  • It will automatically update your documentation.

  • {roxygen2} comments are formatted as tag-value pairs, where tags begin with an ampersand @.

  • Values of a tag extend from the tag to the next tag.

  • A typical {roxygen2} documentation looks like this
    #' @title One line description of what the function does.
    #' 
    #' @description One paragraph description of what the function does
    #'
    #' @details
    #' Long documentation, detailing exactly what the function does
    #' 
    #' @param arg1 What is arg1?
    #' @param arg2 What is arg2?
    #' 
    #' @return What is returned?
    #' 
    #' @author Your name
    #' 
    #' @examples
    #' ## Some example code goes here
    fn <- function(arg1, arg2) {
      ## Function code here
    }

  • @param: Each argument should be documented. You should state

    1. What is the format of the argument (character vector? data frame? numeric matrix?)
    2. What affect the argument has on the function’s behavior.

  • @examples: Include a few lines of example R code. Do not use @example as this expects only one line.

  • @return: What does your function return (numeric vector, character matrix, etc). Describe not just its type but what it is (posterior probabilities, summation, geometric means, etc)

power1.R

In the R script create a function can gets two vectors and applies the operation defined by pwr1().

pwr2 <- function(x, y){
  if (class(x) != "numeric" | class(y) != "numeric") {
    stop("Check the class of your vectors both must be numeric")
  }
  else if (sum(is.na(x) | is.na(y)) !=0){
    stop("There is at least one missing value in your vectors")
  }
  else if (length(x) %% length(y) !=0 | length(y) %% length(x) !=0 ) { warning(" The lengths are not equal")}

  else if (length(x) == length(y) & sum(is.na(x) | is.na(y)) ==0 ){
    df <- tibble::tibble(x, y)
    return(dplyr::mutate(df, Power = pwr1(x,y)))
  }  else { stop("Check your vectors!")
  }
   }
